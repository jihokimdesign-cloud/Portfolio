-- Learners were quietly quitting a VR training platform. I designed the AI layer that brought them back.

TAP3D prepares workers for advanced manufacturing roles across VR and Web. As the sole product designer on a six-person team, I took an AI agent from risky idea to funded roadmap — by simulating it with 24 real users before a single model was wired up.

<SectionTitle num="01" kicker="The Problem" title="Silent churn, module by module" />

Internal data revealed a significant lag in the training completion cycle — directly impacting customer retention metrics.

The quantitative data showed where learners dropped off. It couldn't show why.

I conducted nine in-depth interviews and affinity-mapped 200+ coded quotes around two questions: what triggers a user to start right now, and what's the minimum information needed to commit?

<SectionTitle num="04" kicker="Validation" title="So I became the AI" />

The MVP solved "see the task list" — but not "what should I do first," "I'm stuck," or "what did I do last week." The agent could solve all three, but it existed only in mockups, and mockups don't win budget conversations.

For a **Wizard of Oz simulation with 24 users**, I played the agent's backend in real time — reading uploads, drafting handoffs, recommending next steps — while the team observed real behavior with a product that didn't exist yet.

<Flow
  nodes={[
    { label: "The learner", title: "Uses the “AI agent”", note: "real tasks on the real platform — believes it's live" },

Playing the backend meant every reply the "AI" sent was typed by hand — which is how the recommendation script got tuned line by line. The pattern that survived: one action, one reason, one timebox.

### Decision 01 — The sort order that buried the product

Mid-simulation, the task-start numbers stalled in a strange way: users were adding checklist items but never starting them. Watching sessions live with the PM, we found the culprit — the list sorted oldest-first, an LMS default nobody had questioned, so the task a learner just added landed at the bottom. The core loop — add a task, do the task — was being defeated by a sort order.

<TradeOff
  options={[
    {
      label: "As tested",
      title: "Oldest-first",
      text: "Chronological, the platform default — finish what you started first.",
      verdict: "Users added items, then never started them. Anything new sat buried below the fold.",
    },
    {
      label: "Between rounds",
      title: "Newest-first + “New”",
      text: "The task you just added is the first thing you see, flagged with a “New” indicator.",
      verdict: "Task starts jumped immediately in the next round — part of the 38% → 63% lift.",
      selected: true,
    },
  ]}
/>

This is the argument for simulating with rounds instead of a one-shot test: the flaw was caught, fixed, and re-validated inside the same study — before a line of production code existed.

<SectionTitle num="07" kicker="Reflection" title="What I'd carry forward" />

-- Simulation testing seeded a culture of data-driven design at TAP3D.

Pre-launch simulation wasn't common there before this project — proving its value to the C-suite changed how the company validates ideas. It also taught me that a phased MVP isn't a compromise; it's the path that makes the ambitious version fundable.
